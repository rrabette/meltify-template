Meltify

This is an underground radio station. The music here sits somewhere between drone, motorik, psych folk, noise and home-recorded experiment — made by people completely inside their own world, indifferent to audiences, indifferent to genre, indifferent to whether anyone is paying attention. You won't find this on Spotify's algorithms. That's the point.

Big Blood have one of the most remarkable catalogues in underground music — twenty years of recordings, constantly evolving, never repeating themselves. They deserve their own curation.

Big Blood started as a duo in Maine in the mid-2000s, then became a family band as their kids grew up playing alongside them. Over two decades they've drifted between psych folk, noise pop, home-recorded pop and pure experiment — never settling, never repeating themselves, always sounding like nobody else. Their trajectory is one of quiet persistence: releasing constantly, touring rarely, completely indifferent to whether anyone was paying attention.

Hit play and it shuffles everything. Two buttons shape your experience — Molt any track you don't want to hear again and it's gone from this curation forever. Melt any track you love and it's added to your own personal curation, which builds up over time across all our curations. Nothing is saved to any server. No login required. It all lives in your browser.

What I'm fundamentally doing is exploring methods to find new and interesting music — and interesting ways to democratise the sharing of it, which I think needs to be reborn. Each curation is a snapshot of where I am in that exploration. When I've got a new one ready, I publish it here. You're welcome to join me on that journey — or go off and do something completely different with it.

There are three ways to engage with Meltify, all at zero cost:

As a listener — play the station, molt any track you never want to hear again, melt any track you love into your own personal curation. Your curation builds up over time across all published curations. Transfer it between your phone and desktop using a simple code — no login, no account required.

As a builder — clone Meltify and build your own station. See BYO for the full guide.

As a developer — build your own scraping tools and methodologies for finding music across thousands of open source archives worldwide. Contact ross@peer.school to get started.

I want to build my own personal curated radio station like meltify.org. Before we start, give me a quick orientation: Supabase is a free hosted database I can read and write from the browser using JavaScript fetch calls over HTTPS — no backend needed. Cloudflare Workers serves a single HTML file as a live website, free, auto-deploys from GitHub on every push. Archive.org has millions of free, legally streamable music tracks with an open API — no hosting costs, no API key required. Once I understand the stack, walk me through building it step by step. Start by asking me what I already have set up, then take me through one step at a time. --- STEP 1 — SUPABASE Create a free Supabase project at supabase.com (sign in with GitHub). Once created, go to Settings → Data API and copy the Project URL and anon/public key — you will need both in every step. Run the following SQL in the Supabase SQL editor to create four tables: Table: tracks Columns: id (bigserial primary key), title (text), artist (text), album (text), url (text), duration (text), artwork (text), source (text), license (text) Table: melts Columns: id (bigserial primary key), user_id (text), track_url (text), artist (text), title (text), album (text), artwork (text), melted_at (timestamptz default now()) Table: molts Columns: id (bigserial primary key), track_url (text), artist (text), molted_at (timestamptz default now()) Table: user_curations Columns: id (bigserial primary key), user_id (text), track_url (text), artist (text), title (text), album (text), artwork (text), added_at (timestamptz default now()) After creating the tables, go to Authentication → Policies and disable Row Level Security on all four tables so the browser can read and write without a login. This is safe because no personal data is stored. --- STEP 2 — GITHUB AND CLOUDFLARE WORKERS Create a new empty GitHub repository. Create two files: index.html (the player) and config.js (credentials). In config.js write: const SUPABASE_URL = 'your-project-url'; const SUPABASE_KEY = 'your-anon-key'; const ADMIN_PASSWORD = 'choose-a-password'; In Cloudflare, go to Workers & Pages → Create → Worker. Paste the following worker script which serves index.html: export default { async fetch(request, env) { const url = new URL(request.url); if (url.pathname === '/config.js') { return new Response(env.CONFIG_JS, { headers: { 'content-type': 'application/javascript' } }); } return new Response(env.INDEX_HTML, { headers: { 'content-type': 'text/html' } }); } } Store your file contents as Worker environment variables named INDEX_HTML and CONFIG_JS. Connect your custom domain in the Worker settings. Every time you update the files, update the environment variables and redeploy. --- STEP 3 — THE PLAYER (index.html) This is a single self-contained HTML file. No frameworks, no build step. Design system: pure black background #000000. Font: JetBrains Mono throughout — import from Google Fonts. Accent colour: #1DB954. Muted text: #6b6b6b. All panels are full-page replacements of the main player — nothing overlaps. Layout — mobile first, single column: ARTWORK AREA: full width, fixed height 260px on mobile, square aspect ratio on desktop. The artwork image is a CSS background-image, cover-fit, centred. Over it: a dark gradient scrim — transparent from 0% to 60%, then rapid drop to rgba(0,0,0,0.92) at 100% — sits between the image and the text overlay. Track title (white, 15px, bold), artist (green, 12px), album (grey, 11px) are absolutely positioned bottom-left of the artwork, over the scrim. A small ⓘ icon sits top-right of the artwork — 32px circle, semi-transparent black background — pulses with a soft green glow animation every 6 seconds (CSS keyframes). Tapping it opens the info overlay. PROGRESS BAR: below artwork, full width, 6px height, white fill on dark grey track, clickable anywhere to seek. Timestamps (current time / total duration) sit either side below the bar in 11px grey mono text. CONTROLS ROW: centred, three elements — prev · play/pause · skip. Play/pause is a 52px white circle with a black play or pause icon inside. Prev and skip are plain borderless text buttons, same 13px font size, grey colour. All three are vertically centred on the same row. MELT / MOLT ROW: directly below controls, two buttons side by side — melt and molt — same size as prev and skip, borderless text. Melt is green, molt is grey. INLINE FEEDBACK LINE: a single line of text directly below the melt/molt row. Hidden by default. On melt: fades in "melted into your curation ✓" in green. On duplicate melt: fades in "already in your curation" in grey. On molt: fades in "track molted ✓" in grey. Each fades out automatically after 1.8 seconds. Never stacks — always replaces previous message. FOOTER: stacked vertically, centred, generous bottom padding (60px minimum for mobile thumb space). Three plain text links: BYO · Curations · Privacy. No border above footer. Footer hides completely when any panel is open. --- STEP 4 — LOADING AND PLAYING TRACKS On page load, fetch all tracks from Supabase. Supabase paginates at 1000 rows — loop using offset until fewer than 1000 rows are returned, then concatenate all results. Remove any tracks whose URL is in the localStorage molt list (key: meltify_molted, stored as JSON array of URLs). Shuffle the remaining tracks using Fisher-Yates. Store the shuffled array in memory. Do not call audio.play() on load — browsers block autoplay without a user gesture. Instead, prepare the first track (set audio.src, preload metadata) but wait for the user to tap play. When a track loads, update: artwork background-image, title, artist, album overlay text, progress bar reset to 0, timestamps reset to 0:00. Auto-skip on stream failure: if audio throws an error or does not start within 2.5 seconds, skip to the next track. Count consecutive auto-skips — stop auto-skipping after 4 in a row to avoid an infinite loop on a bad connection. Reset the counter when a track plays successfully. On track end (audio onended event), advance to next track in the shuffled array. Loop back to start when the end of the array is reached. --- STEP 5 — MELT SYSTEM Melt adds the current track to the user's personal curation. FIRST TIME ONLY: if localStorage key meltify_melt_explained does not exist, show a bottom-sheet modal before doing anything else. The modal slides up from the bottom, dark background, explains: "Melt any track you love. It builds your personal curation — a playlist of everything you've melted, accessible on any device using your MELT code." One button: "Got it — melt this track". On dismiss, set meltify_melt_explained = 1 in localStorage, then proceed with the melt. MELT CODE: if localStorage key meltify_user_id does not exist, generate one now. Format: MELT- followed by 4 random uppercase characters from the set ABCDEFGHJKLMNPQRSTUVWXYZ23456789 (no O, 0, I, 1 to avoid confusion). Store in localStorage as meltify_user_id. Show the code to the user once in the melt modal on first generation: "Your curation code is MELT-XXXX — you only need this if you want to access your curation on another device." DUPLICATE CHECK: before saving, check if the current track URL already exists in localStorage key meltify_curation (JSON array of track objects). If it does, show "already in your curation" feedback and stop. SAVE LOCALLY: add the full track object (url, title, artist, album, artwork) to meltify_curation in localStorage. SAVE TO SUPABASE: POST to the melts table with fields: user_id (the MELT code), track_url, artist, title, album, artwork. This is a fire-and-forget call — use try/catch and fail silently if offline. Also POST to user_curations table with the same fields plus added_at timestamp. SHOW FEEDBACK: show "melted into your curation ✓" in green. --- STEP 6 — MOLT SYSTEM Molt removes the current track permanently from this device's shuffle. FIRST TIME ONLY: if localStorage key meltify_molt_explained does not exist, show a bottom-sheet modal explaining: "Molting a track removes it from your radio forever on this device. It will never play again." One button: "Got it — molt this track". On dismiss, set meltify_molt_explained = 1, then proceed. SAVE LOCALLY: add the track URL to localStorage key meltify_molted (JSON array of URLs). Remove the track from the current in-memory shuffled array immediately so it cannot play again this session. SAVE TO SUPABASE: POST to the molts table with track_url and artist. Fire-and-forget, silent fail. SKIP TO NEXT: immediately advance to the next track. SHOW FEEDBACK: show "track molted ✓" in grey. Molted tracks are never synced between devices — the molt list lives only in the browser where it was created. --- STEP 7 — INFO OVERLAY Opened by tapping the ⓘ icon on the artwork. Full screen, position fixed, z-index above everything, scrolls internally. Resets scroll to top every time it opens. Dark background #0a0a0a. Back button top-left: "← listen" — green, pill shaped border, closes the overlay and returns to the player. Content sections (vertically stacked, generous padding): 1. Expands to reveal: why you built this station, your philosophy around music discovery, and three levels of engagement available: listener (just listen), builder (build your own station using the BYO guide), developer (access the canvas tools to build scraping instruments and publish exhibitions). --- STEP 8 — CURATIONS PANEL Opened by tapping Curations in the footer. Password protected — on load show a password prompt, check against the ADMIN_PASSWORD from config.js, only reveal the panel on correct entry. SEARCH: text input for artist name, Search button. On search, call the Archive.org advanced search API: https://archive.org/advancedsearch.php?q=creator:"ARTIST NAME" AND mediatype:audio&fl[]=identifier,title,creator&rows=50&output=json Show results as cards: identifier, title, creator, checkbox. Select all checkbox at top. Batch Add Selected button. BATCH ADD: for each selected identifier, fetch https://archive.org/metadata/IDENTIFIER/files — find all MP3 files (prefer source:original over source:derivative). For each MP3, fetch https://archive.org/metadata/IDENTIFIER to get title, creator, description. Insert a row into the tracks table via Supabase REST API. Show progress count as it runs. URL PASTE: a second input where you can paste any archive.org/details/IDENTIFIER URL directly and add all its tracks in one click. MANAGE TRACKS: shows all current tracks from Supabase in a scrollable list, filterable by artist or title text input. Each track has a Remove button (deletes from Supabase). Remove All Filtered button removes all matching the current filter. Changes are immediate — no separate save step. --- IMPORTANT GOTCHAS Never open index.html directly from the file system (file:// protocol) — fetch calls to Supabase and Archive.org will be blocked by the browser's CORS policy. Always test through a local server (npx serve .) or your deployed Cloudflare URL. Supabase paginates results at 1000 rows. If your playlist has more than 1000 tracks, you must loop: fetch with range 0-999, then 1000-1999, and so on until a response returns fewer than 1000 rows. Browsers block audio autoplay. Never call audio.play() on page load. Wait for a user gesture (tap on play button) before playing anything. Archive.org streams are unreliable. Always implement the 2.5-second timeout auto-skip. Cap consecutive auto-skips at 4 to prevent infinite loops on a bad connection. Favicon files are aggressively cached. When you change the favicon, add ?v=2 to the link href in the HTML. The Supabase anon/public key is safe to include in client-side JavaScript — it is designed to be public. Never include the service role key in client code. For the MELT code generator, exclude the characters O, 0, I, and 1 from the character set to prevent visual confusion when users read or type their code. Download

Install Meltify as an app on your device — no app store needed. It runs directly from the browser and works like a native app once installed.

On Desktop

Open meltify.org in Chrome or Edge. Look for the install icon in the address bar — it looks like a small screen with a down arrow. Click it and select Install. Meltify will appear in your taskbar and open as a standalone app.

On Android

Open meltify.org in Google Chrome — not the built-in browser. Tap the three-dot menu and select Add to Home Screen. Meltify will appear on your home screen like any other app.

On iPhone

Open meltify.org in Safari. Tap the share icon at the bottom of the screen — the square with an arrow pointing up. Scroll down and tap Add to Home Screen. Meltify will appear on your home screen.

Privacy

Nothing here knows who you are.

Molted tracks stay in your browser and never leave. Your curation builds up as you listen. It lives in your browser — no account needed, no code to remember. If you ever want to access your curation on a different device, go to Curations to find your code and enter it there.

No cookies. No tracking. No analytics. We couldn't identify you from what we store even if we wanted to.

We use your browser's localStorage — not cookies. localStorage is a private storage area inside your browser that no external server can read remotely. Nothing stored there travels anywhere unless you choose to send it. Your molted tracks never leave your device. When you melt a track, your browser sends the track details and your anonymous MELT-XXXX code to our database over HTTPS — an encrypted connection. Nothing else is sent. No name, no email, no IP address, no identity of any kind.

— Your MELT-XXXX code

What goes to our database

Only if you melt a track — the track URL, artist name, title, and your anonymous MELT-XXXX code. Nothing else. The code is randomly generated in your browser and is not linked to any identity. We never see your name, email, IP address or any device information.

GDPR

Because no personally identifiable information is collected at any point, GDPR technically does not apply. A randomly generated code connected to a list of track URLs is not personal data. There is no way to link what we store to any individual.
